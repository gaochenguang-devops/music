# Sonora 架构设计

## 线程与数据流

UI 线程持有播放列表、歌词、配置和瞬时视图状态。音频线程独占 `MixerDeviceSink` 与 `rodio::Player`，避免底层设备对象跨线程共享。UI 使用 `std::sync::mpsc` 发送命令；音频线程每 75 ms 回传播放器的真实位置，并在队列变空时发送完成事件。

```text
egui UI ── PlayerCommand ──> 音频线程 ──> rodio / CPAL
egui UI <── PlayerEvent  ─── 音频线程 <── 播放位置/设备状态
   │
   ├── Playlist / ID3 / Symphonia probe
   ├── LRC parser
   └── JSON config
```

输出设备切换是一次受控重建：先记录位置与暂停状态，销毁旧 Player 和输出流，在选中设备上创建新流，重新加载当前文件，最后恢复音量、倍速、位置和播放状态。

## 模块边界

`player` 不认识播放列表或播放模式，它只执行单首音频命令。上一曲、下一曲、循环和随机策略由 UI 协调层决定，因此播放核心保持简单且可替换。`playlist` 负责磁盘对象变成 `Track` 的全过程，包括 ID3、封面、LRC 路径和时长。`lrc` 是纯解析模块，不依赖 GUI 或播放库，能够独立测试。

播放由 rodio 0.22 完成；MP3 的容器探测和时长计算直接使用 symphonia 0.6。rodio 自身也以 symphonia MP3 feature 解码，从而在播放控制和格式解析之间保持清晰分工。

## 错误处理

磁盘读取和元数据探测使用 `anyhow::Result` 逐层添加上下文。音频线程不能把设备错误直接返回到 UI 调用栈，因此转换为 `PlayerEvent::Error`，由界面显示可关闭的中文提示。批量导入不会因单个损坏文件中断，其错误会汇总并显示首个原因。

## 配置

`directories::ProjectDirs` 选择平台规范的配置目录，`serde_json` 保存音量、播放模式、倍速、主题和设备名称。配置写入失败不影响播放器退出；无效或旧配置会回退到默认值。

## 测试策略

纯逻辑测试覆盖 LRC 标签、重复时间戳、同步定位、文件名兜底和长时长格式。完整构建检查跨模块 API 契约。音频设备和原生窗口依赖当前机器硬件，因此通过实际启动程序进行冒烟验证。
